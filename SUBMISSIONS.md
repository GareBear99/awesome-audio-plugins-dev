# Submissions

A practical guide for suggesting new plugins, bundles, frameworks, DSP resources, and related audio-tooling pages for this repository.

This repository is intentionally curated. The goal is not to accept everything. The goal is to keep the list useful, readable, credible, and worth returning to.

---

## What can be submitted

Submissions are welcome for:

- free audio plugins
- free plugin bundles
- open-source DSP plugin repositories
- plugin-development frameworks and APIs
- DSP learning resources
- free sample packs and sound resources
- repo improvements such as structure, wording, broken links, or category cleanup

---

## What usually gets accepted

A submission should usually satisfy at least one of these:

- it is a strong free plugin with a real-world use case
- it is an open-source DSP or plugin codebase worth studying
- it is a framework, API, or standard relevant to plugin development
- it is a learning resource with lasting technical value
- it is a sound resource that fits naturally into the wider audio-tooling ecosystem

Strong submissions are usually:

- clearly free
- easy to verify
- relevant to producers, plugin developers, or DSP learners
- not obvious duplicates of entries already listed
- useful enough to deserve attention without bloating the repo

---

## What usually does not get accepted

These are common rejection reasons:

- paid-only plugins
- weak “top 500 freebies” style dump pages
- abandoned or broken links with no practical value
- duplicate entries already covered elsewhere in the repo
- low-information submissions with no clear reason to exist here
- tools that are unrelated to audio plugins, DSP, or adjacent sound-resource workflows

---

## Before submitting

Please check these first:

1. Is it already in the repo?
2. Is the project clearly free?
3. Is there a direct official link?
4. Is it useful enough to help someone discover, learn, build, or ship?
5. Does it fit an existing category?

If the answer to most of those is no, it probably should not be submitted.

---

## Preferred submission format

Use a short, direct format.

```md
**Name:** Plugin / project name
**Link:** Official URL
**Category:** Equalizer / Synth / Bundle / Framework / DSP Resource / etc.
**Why it belongs:** 1–3 sentences
**Free status:** Fully free / open-source / free tier / legacy free release
```

Example:

```md
**Name:** ExamplePlugin
**Link:** https://example.com
**Category:** Reverb
**Why it belongs:** Free algorithmic reverb with a real production use case and a clear official download page.
**Free status:** Fully free
```

---

## Good submission examples

### Good
- a respected free synth with an official homepage
- an open-source DSP repo with educational value
- a plugin framework with current relevance
- a strong free utility or metering plugin that fills a real gap

### Weak
- random mirror link with no official source
- listicle page that only republishes other people’s work
- vague “this is cool” suggestion with no category or reason
- duplicate of something already listed in a stronger category page

---

## Where to submit

### Option 1: GitHub Issue
Use an issue when you want to suggest:
- a new entry
- a category correction
- a broken link fix
- a cleanup pass
- a wording improvement

Suggested issue title styles:
- `Add: Plugin Name`
- `Fix: broken link for Plugin Name`
- `Category suggestion: Add Bundle Name to bundles page`
- `Cleanup: remove duplicate entry`

### Option 2: Pull Request
Use a pull request when you already know where the entry belongs and can add it cleanly.

Best for:
- adding an entry with clear formatting
- fixing links
- improving wording
- adding a focused new category page

---

## Style rules for accepted entries

Keep entries:

- short
- specific
- readable at a glance
- non-hype
- non-spammy

Preferred style:

```md
* [Plugin Name](https://example.com) — One-line description with clear practical value.
```

Avoid:

- giant marketing blurbs
- exaggerated claims
- keyword stuffing
- multi-paragraph plugin descriptions inside list sections

---

## Category fit guide

Use the closest matching category:

- **Equalizers** for EQs, dynamic EQs, and filter/EQ hybrids that are primarily EQ tools
- **Synthesizers** for instruments and synth engines
- **Compression** for compressors, multiband compression, and dynamics tools
- **Saturation / Distortion** for drive, tape, waveshaping, clipping, or analog coloration
- **Reverb** for space and ambience processors
- **Delay** for echo, feedback, and delay-focused tools
- **Creative Effects** for glitch, stretch, vinyl, texture, and unusual processors
- **Modulation** for chorus, phaser, flanger, and related tools
- **Filters** for dedicated filter tools
- **Utility / Metering** for analyzers, loudness meters, and production utilities
- **Open-Source DSP Plugins** for codebases worth studying
- **Plugin Bundles** for multi-plugin collections
- **Plugin Frameworks** for developer tooling and standards
- **DSP Resources** for books, archives, and technical learning references
- **Sample Packs and Sound Resources** for sound libraries and related resources

If an entry could fit multiple sections, choose the one where it is most useful to the reader.

---

## Editorial preference

This repository prefers:

- stronger entries over bigger counts
- official sources over third-party mirrors
- practical usefulness over novelty
- readable pages over inflated pages
- credibility over hype

A smaller list with better signal is better than a huge list full of filler.

---

## Suggested review checklist

Before accepting a submission, verify:

- official source is available
- free status is clear
- category is correct
- description is concise
- link works
- entry is not already represented elsewhere in a better way
- the addition improves the repo instead of just making it longer
